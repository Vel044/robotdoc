# 实验 06 设计文档：摄像头感知层级耗时剖析

## 一、测量方法总览

由于摄像头链路跨越 **Python 主线程 / Python 后台线程 / OpenCV C++ / Linux 内核** 四个执行域，
单一计时手段无法覆盖全部 9 层。本实验采用**双层时间戳**：

| 域 | 工具 | 时间戳分辨率 |
|---|---|---|
| 内核态（pselect6, ioctl） | ftrace tracepoint（`sys_enter_*` / `sys_exit_*`） | 纳秒 |
| Python 用户态（线程内 + 线程间） | `time.perf_counter_ns()` | 纳秒 |

两套时间戳通过 **PID + 时间戳对齐**：
- ftrace 输出包含 `pid`、`ts`、`event`；
- Python 在每帧采集前后记录 `pid`、`tid`、`perf_counter_ns()`；
- 后处理脚本以 `pid` 过滤 ftrace 中属于 LeRobot 进程的事件，按时间戳归并到对应帧。

## 二、Python 端插桩（layer 3–9）

不修改主代码库，通过 monkey-patch 在 `profile_camera.py` 启动时切入：

```python
# 简化伪代码
from lerobot.cameras.opencv.camera_opencv import OpenCVCamera

_orig_read = OpenCVCamera.read
_orig_postprocess = OpenCVCamera._postprocess_image
_orig_async_read = OpenCVCamera.async_read

def patched_read(self):
    t0 = time.perf_counter_ns()
    grab_ok = self.videocapture.grab()       # 内部含 ioctl(VIDIOC_DQBUF) [layer 2]
    t1 = time.perf_counter_ns()
    ret, frame = self.videocapture.retrieve()  # libjpeg 解码 + ndarray 构造 [layer 3+4]
    t2 = time.perf_counter_ns()
    record_layer(self.episode, self.frame_idx, "grab", t1 - t0)
    record_layer(self.episode, self.frame_idx, "retrieve_decode_alloc", t2 - t1)
    return ret, frame

def patched_postprocess(self, frame):
    t0 = time.perf_counter_ns()
    out = _orig_postprocess(self, frame)
    record_layer(self.episode, self.frame_idx, "postprocess_bgr2rgb", time.perf_counter_ns() - t0)
    return out

def patched_async_read(self, timeout_ms=200):
    t0 = time.perf_counter_ns()
    out = _orig_async_read(self, timeout_ms)
    record_layer(self.episode, self.frame_idx, "async_read_lock_copy", time.perf_counter_ns() - t0)
    return out
```

主线程预处理（layer 7–9）通过包装 `RobotProcessor` 的 `__call__` 实现，
在 `observation_processor` / `normalize_processor` / `device_processor` 各包一层 `perf_counter`。

注意：**`videocapture.grab()` 与 `retrieve()` 必须分开调用以拆出 layer 2 / 3+4**。
原代码用 `videocapture.read()` 是 `grab()+retrieve()` 的合写；分开调用后单帧测量行为应严格等价
（OpenCV 对此明确支持，见 `cv::VideoCapture::read` 实现）。

## 三、内核端插桩（layer 1, 2）

### 3.1 启用 tracepoint

```bash
# start_ftrace.sh
echo > /sys/kernel/debug/tracing/trace
echo 1 > /sys/kernel/debug/tracing/events/syscalls/sys_enter_pselect6/enable
echo 1 > /sys/kernel/debug/tracing/events/syscalls/sys_exit_pselect6/enable
echo 1 > /sys/kernel/debug/tracing/events/syscalls/sys_enter_ioctl/enable
echo 1 > /sys/kernel/debug/tracing/events/syscalls/sys_exit_ioctl/enable
echo 16384 > /sys/kernel/debug/tracing/buffer_size_kb
echo 1 > /sys/kernel/debug/tracing/tracing_on
```

### 3.2 过滤 ioctl 命令

`VIDIOC_DQBUF` 的 cmd 值依赖架构编译，可通过 `grep VIDIOC_DQBUF /usr/include/linux/videodev2.h`
查得宏定义后用 `_IOWR('V', 17, struct v4l2_buffer)` 计算：

```c
// _IOWR(type, nr, size) = (3<<30) | (size<<16) | (type<<8) | nr
// 在 64-bit ARM 上 sizeof(struct v4l2_buffer) = 88
// VIDIOC_DQBUF = 0xc0585611
```

ftrace `set_ftrace_filter` 不支持 cmd 过滤；后处理脚本读取 `arg[1]`（ioctl 第二个参数）按
`0xc0585611` 过滤。

### 3.3 PID 过滤
启动 LeRobot 后获得主进程 PID，写入 `set_event_pid`：
```bash
echo $LEROBOT_PID > /sys/kernel/debug/tracing/set_event_pid
```
注：摄像头后台线程是 LeRobot 主进程内的线程，共享 PID，无需额外过滤 TID。

### 3.4 导出与停止

```bash
# stop_ftrace.sh
echo 0 > /sys/kernel/debug/tracing/tracing_on
cp /sys/kernel/debug/tracing/trace /tmp/lerobot_ftrace_$(date +%s).log
echo 0 > /sys/kernel/debug/tracing/events/syscalls/sys_enter_pselect6/enable
echo 0 > /sys/kernel/debug/tracing/events/syscalls/sys_exit_pselect6/enable
echo 0 > /sys/kernel/debug/tracing/events/syscalls/sys_enter_ioctl/enable
echo 0 > /sys/kernel/debug/tracing/events/syscalls/sys_exit_ioctl/enable
```

## 五、可能的偏差源与对策

| 偏差源 | 影响 | 对策 |
|------|------|------|
| 主线程在 `async_read` 拿锁时正好碰上后台线程写帧 | layer 6 偶尔尖刺 | 取均值（n=90），上报 P50 / P95 |
| `perf_counter` 调用本身耗时（约 50–100 ns） | layer 4、5、9 等亚毫秒层占比偏高 | 单独测一组"空 monkey-patch"baseline，从所有层减去 |
| ftrace 导出阶段 ring buffer 覆盖 | 长 episode 后期 layer 1、2 数据丢失 | 限制每 episode 30 帧；buffer 设到 16 MB |
| 摄像头线程被绑核到大核（`taskset`） | layer 3 libjpeg 解码偏低 | 不绑核，让 CFS 自然调度，与生产配置对齐 |
| 两个摄像头 USB 总线竞争 | layer 1 偶尔抖到 50 ms | 按 `device_id` 分组统计，不混合 |

## 六、验收标准

- A 组（无 ftrace）和 B 组（开 ftrace）在 layer 3–9 的 P50 差距 < 5%；
- 各 episode 90 帧中至少 85 帧成功记录全 9 层（缺帧率 < 6%）；
- 图 2 上"端到端 (1+2+3+4+5)"参考线 ≈ 33 ms ± 5 ms；
- 图 2 上"主线程感知 (6+7+8+9)"参考线 ≈ 2–3 ms；
- 与已有文档（`Lerobot相机调用链分析.md` 中的"BGR→RGB 0.1–0.3 ms"等定性结论）一致。
