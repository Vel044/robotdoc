# 实验设计：实验环境与测量工具校准

## 1. 实验定位

这个实验服务第 2 章，不服务第 4 章优化。它要补的是“实验方法可信度”：同一条 `record_loop` 路径，在不开 tracing、开 `strace`、开 `ftrace`、实时导出 trace、使用 `trace-cmd` 时，主循环会不会被测量工具明显扰动。

第 2 章里可以把它放在 2.5 工具链选型之后，作为 `strace` 和 `ftrace` 取舍的实测依据；2.6 的硬件软件配置则由本实验的环境快照固化。

## 2. 固定条件

| 项目 | 固定值 |
|---|---|
| 推理主机 | 树莓派5，ARM Cortex-A76，4 核，4 GB LPDDR4X，无 GPU |
| 系统 | Raspberry Pi OS |
| 机器人 | SO-101，在线推理使用 1 个 follower，6 个舵机 |
| 摄像头 | 2 路 OpenCV USB 摄像头，640 x 360，30 fps |
| record 参数 | `chunk_size=100`，目标 `fps=30`，episode 约 30 s |
| 任务 | 固定一个代表任务，建议使用抓取任务 |
| 重复次数 | 预实验每组 3 次；正式实验每组 5 次 |
| 环境 | 同一供电、同一 USB 连接、同一模型权重、同一 camera index |

正式实验前先空跑 1 个 episode，排除冷启动、模型首次加载和摄像头首次初始化的影响。后续每组之间间隔 30--60 s，让树莓派温度回到接近状态。

## 3. 实验分组

### A 组：baseline

不开任何 tracing。只保留 LeRobot 自身阶段计时日志，并用 `/usr/bin/time -v` 包住 record 命令。

这组回答：在最接近真实运行的条件下，FPS、阶段占比、CPU 时间和内存占用是多少。

### B 组：strace

用 `strace -f -ttT -o raw/strace_RUN.log` 包住完全相同的 record 命令。

这组不追求“准确性能”，而是故意量化 `strace` 的扰动。重点看 `system_time_s`、上下文切换次数和 FPS 是否明显变坏。只要这组明显偏离 baseline，就能支撑论文里的判断：`strace` 适合前期定性探索，不适合最终定量计时。

### C 组：ftrace 延迟导出

episode 开始前同样要打开所需 tracepoint，并把 `tracing_on` 置为 1。区别是 episode 运行期间不启动任何用户态读日志进程，只让事件写入内核 ring buffer；episode 结束后先把 `tracing_on` 置为 0，再复制 `/sys/kernel/debug/tracing/trace`。

这组回答的是：ftrace 插桩本身是否足够轻。运行期间只有被测进程和内核 trace 写入，没有额外的 `cat` 进程、文件写入和导出 I/O。如果 C 组接近 baseline，说明正式实验可以使用 ftrace 作为内核态时间来源。

代价是 ring buffer 有容量上限。事件太密或 episode 太长时，旧事件可能被覆盖，最后复制出来的 `trace` 只剩较新的尾部。所以 C 组要么把 buffer 调大，要么控制 episode 长度。

### D 组：ftrace 实时导出

episode 开始前也要打开 tracepoint 和 `tracing_on`，但还要在 record 命令运行前启动一个后台导出进程，例如持续读取 `trace_pipe` 并写入 `raw/ftrace_D_RUN.log`。这个后台进程会在 episode 期间不断发生 `read()`、`write()`、调度和文件 I/O。

这组回答的是：实时导出是否把额外压力带进了实验。它比 C 组多出来的不是 ftrace 插桩，而是用户态导出链路。重点看 FPS、系统时间、RSS、上下文切换、日志大小和 lost events。若 D 组明显变差，后续正式实验应避免边跑边导出。

注意：如果 episode 跑完才开始读 `trace_pipe`，那就不是 D 组了；如果 ring buffer 已经覆盖，只会读到很短一段尾部数据。D 组正确顺序是“配置 tracepoint -> 清空 buffer -> 打开 `tracing_on` -> 启动实时导出 -> 启动 record -> record 结束 -> 停止导出 -> 关闭 `tracing_on`”。

### E 组：trace-cmd record

用 `trace-cmd record` 记录与 C/D 组相同的 tracepoint。

这组回答：`trace-cmd` 能不能替代手写 ftrace 导出流程。它通常更适合长时间采集，因为它围绕 per-CPU buffer 和 trace.dat 做了更完整的封装。

## 4. 建议 tracepoint

本实验不需要把所有内核细节都打开，只打开和主循环等待、摄像头、串口 I/O 直接相关的事件：

| tracepoint | 作用 |
|---|---|
| `syscalls:sys_enter_ioctl` / `syscalls:sys_exit_ioctl` | 观察摄像头 V4L2 ioctl 和串口 ioctl |
| `syscalls:sys_enter_pselect6` / `syscalls:sys_exit_pselect6` | 观察 OpenCV/V4L2 等待设备事件 |
| `syscalls:sys_enter_read` / `syscalls:sys_exit_read` | 观察串口和文件读取 |
| `syscalls:sys_enter_write` / `syscalls:sys_exit_write` | 观察串口写入 |
| `syscalls:sys_enter_clock_nanosleep` / `syscalls:sys_exit_clock_nanosleep` | 观察 `fps_sleep` / `busy_wait` 中可能进入内核的睡眠 |
| `syscalls:sys_enter_futex` / `syscalls:sys_exit_futex` | 观察 Python 线程事件、锁竞争和条件等待 |

如果事件量太大，优先保留 `ioctl`、`pselect6`、`clock_nanosleep` 和 `futex`。本实验的目标是校准工具开销，不是完整复原每一帧的调用链。

## 5. 执行流程

### 5.1 记录环境快照

正式跑实验前，先按 `environment_snapshot_template.md` 填一份环境快照。至少记录：

- 操作系统版本、内核版本、Python 版本；
- LeRobot、PyTorch、torchvision、OpenCV、NumPy、pyserial、feetech-servo-sdk 版本；
- `/dev/video*`、`/dev/ttyACM*` 设备节点；
- 摄像头分辨率、像素格式、目标 fps；
- 树莓派温度、CPU 频率、供电状态；
- 模型权重路径、任务名、`chunk_size`、目标 `fps`。

### 5.2 每次 episode 的记录顺序

每个 run 都按同一个顺序记录：

1. 记录开始温度和 CPU 频率。
2. 启动对应组别的 tracing。
3. 用 `/usr/bin/time -v` 包住同一条 record 命令。
4. 停止 tracing。
5. 记录结束温度和 CPU 频率。
6. 从 LeRobot 日志提取 `fps_mean` 和四阶段占比。
7. 把结果填入 `tool_overhead_summary_template.csv`。

### 5.3 命令模板

下面的 `<record_loop 命令>` 用真实的 LeRobot record 命令替换。五组实验必须使用同一条命令，除了 tracing 包装不同，其他参数不要变。

baseline：

```bash
/usr/bin/time -v -o raw/time_A_RUN.txt <record_loop 命令>
```

strace：

```bash
/usr/bin/time -v -o raw/time_B_RUN.txt \
  strace -f -ttT -o raw/strace_B_RUN.log \
  <record_loop 命令>
```

ftrace 延迟导出：

```bash
# 启动前：清空 trace，打开 tracepoint，打开 tracing_on
# episode 期间不要读 trace / trace_pipe
/usr/bin/time -v -o raw/time_C_RUN.txt <record_loop 命令>
# 结束后：先关闭 tracing_on，再复制 /sys/kernel/debug/tracing/trace
```

ftrace 实时导出：

```bash
# 启动前：清空 trace，打开 tracepoint，打开 tracing_on
# record 启动前先开后台导出：
# cat /sys/kernel/debug/tracing/trace_pipe > raw/ftrace_D_RUN.log &
/usr/bin/time -v -o raw/time_D_RUN.txt <record_loop 命令>
# 结束后：停止后台 cat，再关闭 tracing_on
```

trace-cmd：

```bash
/usr/bin/time -v -o raw/time_E_RUN.txt \
  trace-cmd record -o raw/trace_E_RUN.dat \
  -e syscalls:sys_enter_ioctl -e syscalls:sys_exit_ioctl \
  -e syscalls:sys_enter_pselect6 -e syscalls:sys_exit_pselect6 \
  -e syscalls:sys_enter_read -e syscalls:sys_exit_read \
  -e syscalls:sys_enter_write -e syscalls:sys_exit_write \
  -e syscalls:sys_enter_clock_nanosleep -e syscalls:sys_exit_clock_nanosleep \
  -e syscalls:sys_enter_futex -e syscalls:sys_exit_futex \
  -- <record_loop 命令>
```

## 6. 数据表

汇总表使用 `tool_overhead_summary_template.csv`。一行代表一个 episode。最少要填这些列：

| 列名 | 含义 |
|---|---|
| `group` | A/B/C/D/E |
| `run` | 第几次重复 |
| `episode_s` | episode 墙钟时间 |
| `fps_mean` | LeRobot 统计的平均 FPS |
| `obs_pct` / `inference_pct` / `action_pct` / `wait_pct` | 四阶段占比 |
| `user_time_s` / `system_time_s` | `/usr/bin/time -v` 输出 |
| `max_rss_kb` | 峰值 RSS |
| `voluntary_cs` / `involuntary_cs` | 自愿/非自愿上下文切换 |
| `trace_file_mb` | trace 文件大小 |
| `lost_events` | 丢事件数量，没有则填 0 |
| `cpu_temp_start_c` / `cpu_temp_end_c` | 开始/结束温度 |
| `cpu_freq_mean_mhz` | episode 期间平均 CPU 频率 |
| `notes` | 异常情况，例如摄像头掉帧、串口重连 |

## 7. 分析方法

以 A 组 baseline 为 100%，计算各组相对变化：

- `fps_drop_pct = (fps_A_mean - fps_group_mean) / fps_A_mean`
- `system_time_ratio = system_time_group / system_time_A`
- `rss_ratio = max_rss_group / max_rss_A`
- `phase_drift_pp = phase_pct_group - phase_pct_A`

画四张图即可支撑第 2 章：

1. 各组 FPS 相对 baseline 的下降比例。
2. 各组 `user_time_s` 和 `system_time_s` 对比。
3. 各组上下文切换次数对比。
4. 各组阶段占比漂移，重点看 `obs_pct` 和 `wait_pct` 有没有被 tracing 改变。

## 8. 验收标准

建议把正式采集方案定为 C 组或 E 组，前提是满足：

- 相对 baseline，`fps_mean` 下降不超过 5%；
- `obs_pct`、`inference_pct`、`action_pct`、`wait_pct` 任一漂移不超过 3 个百分点；
- `system_time_s` 没有接近 `strace` 组那种数量级放大；
- trace 日志无明显 lost events；
- 温度和 CPU 频率没有显示降频导致的系统性偏差。

如果 D 组变差，而 C/E 组正常，论文里就可以明确写：ftrace 插桩本身不是主要问题，实时导出才是额外开销来源；正式实验应采用 episode 后导出或 `trace-cmd record`。

## 9. 写回第 2 章的位置

实验完成后，第 2.5 节可以补一小段实测结论：

> 在相同 episode 和相同 `record_loop` 参数下，`strace` 组系统时间明显上升，FPS 下降，因此只用于系统调用类型的定性确认；`ftrace` 延迟导出组与 baseline 接近，说明内核 ring buffer 插桩对主循环扰动较小；实时导出组的额外开销来自日志导出 I/O，因此本文正式阶段计时采用 ftrace 延迟导出或 `trace-cmd record`。

第 2.6 节则引用环境快照，说明所有实验的硬件、软件包版本、摄像头参数和 CPU 热状态都被记录。
