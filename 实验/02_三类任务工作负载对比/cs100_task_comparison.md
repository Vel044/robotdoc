# 三任务时间占比对比（cs100）

数据来源：`timing_stats.csv`  
比较对象：bottle_pick (cs100) vs bottle_push vs classification，episode 时长已统一至 ~30s，各 n=20  
仅看百分比，屏蔽 episode 总时长差异的干扰。

---

## 汇总表

| 阶段 | pick | push | classification |
|------|------|------|----------------|
| episode 均值 | 30.0s | 30.7s | 30.0s |
| **obs** | 15.4% ± 1.6% | 15.4% ± 1.9% | 16.7% ± 1.9% |
| **inference** | 36.5% ± 2.5% | 34.9% ± 1.2% | 35.9% ± 2.0% |
| **action** | 4.8% ± 0.3% | 4.4% ± 0.3% | 4.4% ± 0.3% |
| **wait** | 43.3% ± 2.3% | **45.2%** ± 2.0% | 42.9% ± 2.2% |

---

## 关键结论

### 1. inference 占比：三者几乎相同（~35-36%）
- pick 36.5% / classification 35.9% / push 34.9%，差距在 2pp 以内
- **验证了之前的修正假设**：episode 时长统一后，三个任务的推理占比趋同
- cs 相同 → 模型体量相同 → 单次推理时延相同 → 推理占比自然接近
- 早期 classification 推理占比偏高（39.6%）是 **episode 长达 60s 导致树莓派 CPU 热降频**的结果，不是模型本身更重

### 2. wait 占比：push 最高（45.2%）
- push 的 wait 比 pick 高约 2pp，比 classification 高约 2.3pp
- push 动作执行时间更长（推/压动作行程大），系统空等比例更高
- **这是三者之间最显著的差异**，瓶颈在执行侧而非推理侧

### 3. obs 占比：三者一致（~15-17%）
- classification 略高（16.7%），可能是观测帧数略多或处理稍慢，但差异不显著

### 4. action（写舵机）占比：三者一致（~4-5%）
- 写动作耗时极稳定，与任务类型完全无关

---

## 建议

| 任务 | 瓶颈 | 优化方向 |
|------|------|---------|
| push | wait 占比最高（执行慢） | 减小 cs（如 cs50），缩短每次 chunk 等待时长 |
| pick | 较均衡 | 当前 cs100 配置合理 |
| classification | 与 pick 几乎一致 | 无明显推理开销，长任务注意散热避免降频 |
