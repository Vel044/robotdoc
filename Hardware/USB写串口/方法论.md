
## 一、先统一一个心智模型：ACT、LeRobot、OS 各在干啥？

### 1. ACT 层：算法 / 模型逻辑

* 代码：PyTorch / Diffusion policy / ACT 模型
* 输入：图像、机器人状态、语言指令
* 输出：`action`（关节位置/速度、Delta pose、夹爪开合等）
* 本质：一堆 **GPU kernel + CPU Python 逻辑** 的组合

所有这些都运行在 **用户态进程**（例如 `python train.py` / `python act_inference.py`）里，由 OS 调度。

---

### 2. LeRobot 控制层：控制循环 & 与机械臂通信

你现在跑的 LeRobot 大概是这样的结构（简化）：

```python
while True:
    obs = get_observation()          # 从摄像头 / 机器人状态读取
    action = policy(obs)             # ACT 推理（GPU+CPU）
    send_to_robot(action)            # 通过 socket/串口/SDK 下发
    state = read_robot_state()       # 反馈，下个循环用
```

这一层干的事情：

* 保持一个 **固定频率的控制循环**（比如 20Hz / 30Hz）
* 负责：

  * 采集传感器（摄像头/关节编码器）
  * 调用模型
  * 通过某种协议（Modbus、TCP、厂商 SDK）发指令给机器人
  * 做简单的安全检查（限幅、超时等）
* 全部都跑在用户态，靠 OS 提供：

  * 线程 / 进程
  * socket / 文件 IO
  * 定时器 / 事件循环

---

### 3. 操作系统层：你真正要研究的东西

**这里是老师希望你“吃透”的地方。**

OS 负责：

* 进程 / 线程调度（你的 Python 进程、其他后台服务）
* 内存管理（GPU 显存映射、page fault、缓存）
* 设备驱动（网卡、USB 控制器、摄像头、机器人控制卡）
* 中断、定时器、上下文切换

对机器人表现的影响主要体现在：

* **控制周期能不能稳定（jitter）**
* **推理 + 控制的总延迟（从摄像头采样 → 机械臂开始动）**
* **IO 延迟（网络包、USB 指令）**
* **资源竞争（同一台机器跑别的任务导致 GPU/CPU 抢资源）**

### Step 0：先画清楚你“现在的实际系统”

你可以用一页纸（或者 mermaid）画出你当前系统的真实链路，类似这样：

1. **传感器链（上行）：**

   * 摄像头：USB / CSI / 网络 → 驱动 → /dev/videoX → Python 读取
   * 机器人状态：控制器 → 以太网 / 串口 → 驱动 → socket / 串口 API → Python

2. **计算链：**

   * Python 主进程
   * 一个主控制循环线程
   * GPU：PyTorch kernel 调度（CUDA stream）
   * 中间的数据在 CPU / GPU / 内存之间怎么走

3. **执行链（下行）：**

   * Python → socket.send / SDK 调用 → OS TCP/IP 栈 → 网卡驱动 → 机器人控制器 → 电机

**目标：画出一个“方框 + 箭头”图，每条箭头上标出：是谁调用谁，跑在哪个线程/进程上。**

### Step 1：确定控制循环的真实结构（代码级）

你需要在 LeRobot/你自己的控制脚本里，找到三个关键点：

1. **观测采集函数**：`get_obs()` / `env.get_obs()` / `robot.get_state()`
2. **policy 调用**：`action = policy(obs)`（里面调用 ACT）
3. **下发动作**：`robot.send_action(action)` / `env.step(action)`

然后你要做两件事：

#### （1）在这三个点加时间戳

伪代码例子（Python）：

```python
import time

while True:
    t0 = time.perf_counter()
    obs = get_obs()
    t1 = time.perf_counter()
    action = policy(obs)
    t2 = time.perf_counter()
    send_to_robot(action)
    t3 = time.perf_counter()

    log(
        t_obs = t1 - t0,
        t_model = t2 - t1,
        t_send = t3 - t2,
        loop = t3 - t0
    )
```

你要收集的数据：

* 每个循环的：

  * 采集耗时 t_obs
  * 模型推理耗时 t_model
  * 指令发送耗时 t_send
  * 全流程耗时 t_loop
* 观察 **均值 / 方差 / 尾部（p95、p99）**

> 这一步其实已经在桥接“ACT ↔ OS ↔ Robot”：任何抖动大部分就是 OS + IO 造成的。

#### （2）确认这些逻辑跑在哪些线程上

* 是否有单线程循环
* 是否用多线程 / 多进程（DataLoader、相机线程、控制线程分离）
* ACT 推理与发送 action 是同一个线程，还是异步？

你要能回答一个问题：

> “从摄像头拿到一帧图片，到这帧对应的 action 发出去，中间被哪些线程 / 进程接力？”

---

### Step 2：引入操作系统视角——把 loop 和 OS 调度关联起来

#### （1）用 OS 工具观察你的进程

在你的机器人机上，用：

```bash
top / htop
ps -L -p <pid>             # 查看线程
strace -p <pid>            # 看系统调用
sudo perf top / perf record
```

你要弄清楚：

* 这个 Python 进程有多少个线程？

  * main thread
  * DataLoader 线程
  * camera / network 线程
* 每个控制循环期间，主要在做什么系统调用：

  * `read` / `recv`（读传感器、socket）
  * `write` / `sendto`（发指令）
  * `poll` / `epoll_wait` / `select`（事件等待）
  * `nanosleep` / `clock_nanosleep`（控制固定频率）
* CPU 使用率、何时发生上下文切换（`cs` 指标）

这一步的目的：

> **把你之前测出来的“t_obs / t_model / t_send”对上 OS 里发生的行为。**

例如：

* t_obs 抖动很大 → `read` 系统调用阻塞时间很不稳定（相机驱动/网络抖动）
* t_model 抖动不大 → 模型推理主要受 GPU 限制，与 OS 关系相对小
* t_send 抖动大 → socket 发送时间/网络栈排队问题
* t_loop 摆烂 → 进程经常被 OS 抢占（高负载 / 低优先级）

### Step 3：做“控制变量实验”来感受 OS 对机器人的真实约束

**这一步非常重要：你要“刻意折磨操作系统”，看机器人行为怎么变。**

#### 实验 1：CPU 压力 + 进程优先级

1. 在同一台机器上开一个无聊负载：

```bash
stress-ng --cpu 8 --timeout 300
```

2. 观察：

   * 控制 loop 时间统计（t_loop）
   * 实际机器人表现：是否抖、是否慢、是否 miss 掉某些动作

3. 然后把你的进程优先级调高：

```bash
sudo renice -n -10 -p <pid>
sudo chrt -r 10 -p <pid>      # 尝试实时调度（如果内核支持）
```

4. 再看 t_loop 和动作表现。

目的：

> 直观看到 “OS 调度策略 / 负载” 对机器人动作流畅性和稳定性的影响。

---

#### 实验 2：网络延迟 / 抖动（如果是网络机器人）

用 `tc` 注入延迟和抖动：

```bash
sudo tc qdisc add dev eno1 root netem delay 50ms 10ms
# 或加丢包、抖动
```

观察：

* t_send 增大多少
* 机器人动作是否“滞后”、“抖动”
* 尤其是多步任务（比如抓取 → 放置），执行节奏是否被打乱

你会得到一个非常具体的认识：

> **“操作系统的网络栈 + 调度设计，为机器人定义了一个‘延迟下限 + 抖动上限’，这就是你的行动自由度的约束”。**

---

#### 实验 3：定时器粒度 & 控制周期

如果你的控制循环是这样写的：

```python
time.sleep(dt)
```

那 `dt` 在 OS 里会受到：

* 内核调度周期（tick）
* 其它线程竞争
* `sleep` 精度

你可以试：

* 使用 `time.sleep` 的简单循环 vs 使用更精确的定时（比如 `clock_nanosleep`、或自己累积时间误差）
* 使用 `nice` / `chrt` 调整调度
* 然后测「目标 30Hz 控制循环」实际的 **频率 & jitter**

这一步会让你把“**控制理论里假设的 fixed Δt**”和“**OS 能实际给你的 jitter Δt**”联系起来。

### Step 4：从“现象 → 模型化” —— 把 OS 影响总结成几个可量化指标

做到这里，你会手头有一堆数据：

* t_obs / t_model / t_send / t_loop 的分布
* 在不同 OS 状态（高负载 / 高优先级 / 注入延迟）下的变化
* 机器人行为的变化（成功率、时间、轨迹平滑性）

你可以从中抽象出几个 **系统级约束指标**，比如：

1. **控制周期 jitter：**

   * `J = std(t_loop)` 或 `max(t_loop) - min(t_loop)`
2. **端到端感知-行动延迟：**

   * `T = t_loop`（或者更细分）
3. **网络 IO jitter：**

   * `J_net = std(t_send)`
4. **CPU 调度延迟：**

   * 使用 `perf sched` 或 `os noise` 工具测出来

然后你再看：

> 某个任务（如抓取小物体）成功率 / 时长 随着这些指标变化的趋势。

你就能说出类似这样的话（这就是你论文 / 报告要的结论）：

* 当控制周期 jitter 超过 X ms 时，抓取成功率从 90% 降到 60%
* 当端到端延迟从 150ms 升高到 300ms 时，移动目标的跟踪能力明显下降
* 当 OS 把控制进程抢占时间片比例超过 Y% 时，机器人表现开始明显不稳定

---

### Step 5：再回到「ACT ↔ OS ↔ LeRobot」的链路叙述

到这一步你就能很清楚地从上到下讲一段故事（也是你老师想听到的那种）：

1. **ACT 只决定“应该做什么动作”（action distribution）**
   但它假设：

   * 控制周期大致稳定
   * 延迟不会太离谱
   * 机器人状态反馈是“准实时”的

2. **LeRobot 把 ACT 的输出接入一个实际控制循环**

   * 这里的实现方式（同步/异步、多线程/单线程）
   * 直接决定了 OS 怎么调度这些计算和 IO

3. **操作系统定义了一个“动作执行的物理舞台”：**

   * 你最多多少 Hz 能可靠地跑控制 loop
   * 延迟的中位数、尾部有多大
   * 在别的任务也在抢 CPU/GPU/网络时，你还能不能保证机器人动作不崩

> 换句话说：**ACT 学的是“理想世界”里的 policy，而 OS 决定了“现实世界”能提供多接近这个理想的执行条件。**
