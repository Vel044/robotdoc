## 论文结构（共7章）

### 第1章 绪论

- 研究背景：协作机器人 + VLA 模型快速发展，软件栈趋于复杂

- 研究目标与贡献：
  1. 建立跨层次性能分析模型
  2. 端到端量化各阶段瓶颈
  3. 给出优化方向依据

### 第4章 跨层次性能分析模型与剖析方法

4.1 分析模型设计  
- 四层性能模型：AI推理框架层 / Python运行时层 / OS内核层 / 硬件I/O层  
- 各层可观测指标定义（op耗时 / GIL争用 / 系统调用次数及耗时 / 传输字节数）

4.2 工具链选型与方法论  
- strace vs ftrace：strace 带来 9× 系统时间放大（20.12s→2.20s），论文中以 ftrace 为准

- Python logging：粗粒度循环计时  
- perf stat：硬件计数器（cache-miss / IPC）  
- 表：工具对比表（粒度 / 开销 / 适用场景）

4.3 实验配置  
- 硬件：树莓派5（ARM Cortex-A76，4核，8GB LPDDR4X）  
- 软件：LeRobot v0.3.4 / PyTorch 2.x / Python 3.10  
- 关键参数：chunk_size=100，fps=30，波特率=1Mbps

### 第5章 系统级性能剖析结果

#### 5.1 感知阶段

**摄像头路径**  
- V4L2 调用链：cv2.read → grabFrame(VIDIOC_DQBUF) → retrieveFrame(MJPEG解码)
- pselect6 行为：后台线程阻塞 23–32ms 等待帧（符合 30fps 物理极限）
- MJPEG 解码耗时：10–30ms/帧（libjpeg CPU密集）
- 图：摄像头路径调用层次图

**舵机路径**  
- sync_read 链路：motors_bus → GroupSyncRead → PortHandler → ttyACM0
- 协议细节：Sync Read 包格式，Present_Position 寄存器地址56
- 耗时：1.2ms/帧，开销可忽略

**所需材料**：现有 pselect6最终结果.md + OpenCV视频流分析.md + observation读取ACM0流程.md

#### 5.2 推理阶段

**整体耗时分布**  
- 单帧推理：2000–4000ms（chunk切换时），chunk内帧：从 _action_queue 取，<1ms  
- 推理占循环时间 >99%

**op 级别分解（需新实验）**  
- ResNet18 特征提取 vs Transformer Encoder/Decoder vs 前后处理
- 表：各 op 耗时占比

**PyTorch 线程行为**  
- futex 分析：64569次/40s，来自 ATen 线程池 + OpenMP 同步
- 线程数：4核 ARM 上的 OpenMP 并行度
- 图：futex 调用者线程分布饼图

**所需材料**：现有 ACT推理.md + futex最终结果.md + **新实验②**（torch.profiler op-level profiling）

#### 5.3 执行阶段

**sync_write 链路**  
- 耗时：1–3ms，无回包，半双工 TTY 总线
- 图：写路径调用链

**异步推理流水线分析**  
- 不 stall 条件推导：N ≥ 2 × T_infer × F
- 当前参数代入：cs=100 < 需要的120，stall 周期 20帧/70帧
- time ensemble 在当前参数下失效分析
- 图：同步 vs 异步时序对比图

**所需材料**：现有 async_inference流水线分析.md + USB串口写路径.md

#### 5.4 全链路综合分析

- 时间预算表：33.3ms/帧 内各阶段占比（感知5.2ms / 推理均摊后 / 执行4.8ms）
- 系统调用分布：pselect6(36191次) vs futex(64569次) vs ioctl
- 图：全链路性能时序甘特图（一帧内各阶段时序）
- **新实验③（可选）**：CPU频率锁定（cpufreq performance governor）对推理峰值的影响

**所需材料**：现有 recordloop每部分时间全部分析.md + 内核数据搜集分析.md

### 第6章 性能瓶颈分析与优化建议

6.1 瓶颈归因  
- 主瓶颈：ACT推理（CPU矩阵乘法，非IO）
- 次瓶颈：MJPEG解码（可替换格式）
- 非瓶颈确认：系统调用本身开销（pselect6 均值421ns，futex均值<1us）

6.2 优化方向分析（无需实现，分析可行性即可）  

| 方向 | 预期收益 | 可行性 | 实现复杂度 |
|------|---------|--------|-----------|
| INT8/FP16 量化 | 推理降 2–4× | 高 | 中 |
| ONNX Runtime 后端 | 推理降 1.5–2× | 高 | 低 |
| 异步推理（gRPC） | 消除 stall，需 T_infer < 1.67s | 需配合量化 | 高 |
| MJPEG→YUYV | 解码降至0 | 高（相机支持） | 低 |
| 异步舵机读（_async_read） | 隐藏 1.2ms | 低收益 | 低 |
| 异步舵机写 | 不可行（总线冲突+安全） | ❌ | N/A |

6.3 软件架构改进方向  
- 推理模块解耦（独立进程，gRPC通信）
- 摄像头格式统一（避免libjpeg路径）
- 为RobotOS设计的内核优化空间（实时调度器、USB驱动优先级）

**所需材料**：现有 motors_bus异步读写分析.md + async_inference分析.md，不需新实验。

## 需要补充的新实验（优先级排序）

| 优先级 | 实验 | 目的 | 工具 | 估计耗时 |
|--------|------|------|------|---------|
| ★★★ | torch.profiler op-level profiling | 填充第5.2节推理分解 | torch.profiler | 1天 |
| ★★★ | 三类任务（抓取/推倒/分类）各跑一次，采集 Python 日志 | 填充第3章负载特征对比 | Python logging | 1天 |

## 论文写作建议

- **图表优先**：评委看的是数据，建议每章至少1张性能对比图或调用链图
- **系统调用部分可精简**：futex/pselect6 源码分析（当前4-5篇文档）压缩进一节，重点放结论而非推导过程
- **三类场景对比是亮点**：任务书明确要求抓取/推倒/分类，这是区分本文与"只分析一个场景"的关键
- **第4章方法论不要轻视**：strace vs ftrace 的对比（9×开销差异）本身就是一个工程贡献
