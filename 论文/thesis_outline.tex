\begin{center}
  {\zihao{-2}\heiti\textbf{LeRobot 平台感知—推理—执行流程的\\[4pt]系统级性能剖析研究}}\\[1.2em]
  {\zihao{3}\heiti\textbf{摘\quad 要}}
\end{center}

\vspace{1em}
\zihao{-4}\songti
\mytodo{正式摘要待实验完成后撰写，约300-500字。}

本文以树莓派5（ARM Cortex-A76，无GPU）为目标平台，对 LeRobot 框架下 ACT 机器人学习策略的感知—推理—执行三阶段时序特征进行系统级剖析。
通过在 Python 应用层、PyTorch 运行时层、Linux 内核层及硬件I/O层的多工具联合测量（Python logging / torch.profiler / ftrace / strace），
端到端量化了各阶段的时间开销，定位了主要性能瓶颈，
并从算法、运行时、系统架构三个维度梳理了可行的优化路径。

实验结果表明，ACT 单次推理耗时 2000--4000 ms，占总循环时间的 99\% 以上，是系统主瓶颈；
观测和执行阶段开销极低（各约 5 ms），系统调用本身的开销可忽略不计，
排除了 I/O 路径为主要瓶颈的假设。
在 chunk\_size=100 配置下，系统实际帧率约 18--19 fps，距 30 fps 目标仍有明显差距，
瓶颈根源在于 ARM CPU 矩阵乘法（GEMM）计算能力不足，而非感知复杂度或任务类型。

\zihao{-4}\rmfamily
\mytodo{English abstract to be finalized after experiments complete, approximately 300 words.}

This paper presents a system-level performance analysis of the ACT (Action Chunking with Transformers) policy
running on a Raspberry Pi 5 (ARM Cortex-A76, no GPU) within the LeRobot framework.
By combining multi-layer profiling tools across the Python application layer, PyTorch runtime layer,
Linux kernel layer, and hardware I/O layer (Python logging, torch.profiler, ftrace, strace),
we quantify the end-to-end time distribution across perception, inference, and execution stages,
identify the primary bottleneck, and systematically evaluate feasible optimization paths.

Results show that a single ACT inference takes 2000--4000\,ms, accounting for over 99\% of the total
loop time, making it the dominant bottleneck.
Perception and execution stages contribute only $\approx$5\,ms each;
system call overhead is negligible, ruling out I/O as a bottleneck.
At chunk\_size=100, the system achieves $\approx$18--19\,fps against a 30\,fps target.
The root cause is insufficient ARM CPU matrix-multiplication (GEMM) throughput,
independent of task visual complexity or object count.

\section{绪论}

\subsection{研究背景与意义}

近年来，协作机器人在工业制造、医疗服务、仓储物流等领域加速落地，对自主操作能力的需求持续增长。
以 RT-2\cite{rt2}、OpenVLA\cite{openvla} 为代表的视觉—语言—动作（Vision-Language-Action, VLA）大模型，
将视觉感知与动作决策统一到端到端网络中，显著提升了机器人在开放场景下的泛化能力。
然而，此类模型的参数量和计算量远超传统控制方法，
如何在资源受限的嵌入式平台上实现实时推理，成为制约其部署的关键瓶颈。

在开源机器人学习框架方面，HuggingFace 推出的 LeRobot\cite{lerobot} 集成了
ACT（Action Chunking with Transformers）\cite{act}、Diffusion Policy\cite{diffusionpolicy}
等模仿学习策略，并提供从数据采集、模型训练到在线推理的完整工具链；
同时配套推出 SO-101 开源六自由度机械臂，实现了软件框架与低成本硬件平台的一体化开源。
Physical Intelligence 提出的 $\pi_0$\cite{pi0} 以流匹配为动作生成核心，
在双臂协作任务上表现出色；斯坦福 Mobile ALOHA\cite{mobilealoha} 以低成本仿生平台
搭配联合学习，实现了全身遥操作控制。
LeRobot 的在线推理循环采用"感知—推理—执行"三阶段紧耦合架构：
每帧依次完成传感器观测采集（\texttt{get\_observation()}）、
策略网络推理（\texttt{select\_action()}）和舵机指令下发（\texttt{send\_action()}），
三个阶段在同一线程内串行执行，任一阶段的性能瓶颈均会直接影响整体帧率。

在实际部署中，本文选用树莓派5（ARM Cortex-A76，4核，4\,GB LPDDR4X）作为边缘推理平台，
搭配 SO101 六自由度机械臂和 USB 摄像头，在无 GPU 的条件下运行 ACT 策略网络。
初步测试表明，单次 ACT 推理耗时约 1500-3000\,ms，而传感器采集和指令下发各仅约 1-15\,ms，
推理阶段占循环总时间的 99\% 以上。
在 chunk\_size=100（即每次推理输出 100 帧动作）的配置下，
系统实际帧率仅约 18--19\,fps，距离 30\,fps 的实时目标仍有明显差距。

在开源硬件与低成本本体方面，多个平台为机器人学习研究提供了可复现的硬件基础。
LeRobot 配套的 SO-101 六自由度机械臂将硬件设计与学习框架深度绑定，
结构件可 3D 打印，成本低廉，是本文实验平台的直接来源；
ROBOTIS 的 OpenManipulator 系列\cite{openmanipulator} 提供配套 ROS2 的开源机械臂，
硬件图纸与代码全开源，适合教学和原型开发；
TurtleBot 系列\cite{turtlebot4} 作为开源移动机器人的事实标准，最新版本与 ROS2 生态深度集成；
UC Berkeley HybridRobotics 实验室开发的 Berkeley Humanoid Lite\cite{berkeleyhumanoidlite} 采用 3D 打印方案，
为小型人形机器人的开源化提供了新思路。

上述性能瓶颈涉及多个系统层次：PyTorch 推理引擎的算子调度、OpenMP 线程池的并行效率、
Linux 内核的系统调用路径以及 USB/串口的硬件 I/O 延迟。
然而，现有的机器人系统性能分析工作多聚焦于单一层次（如模型压缩、内核调度或通信延迟），
缺乏从 AI 框架层到内核层的跨层次联合剖析，
难以回答"瓶颈究竟在哪一层、由什么原因导致、各优化方向的收益几何"等关键问题。

因此，本文的研究意义在于：
（1）理论层面——建立覆盖 AI 推理框架层、Python 运行时层、OS 内核层和硬件 I/O 层的四层性能分析模型，
提出面向"感知—推理—执行"紧耦合场景的多工具联合剖析方法；
（2）实践层面——端到端量化 LeRobot 在嵌入式 ARM 平台上的各阶段时间开销，
定位主要性能瓶颈，系统评估 INT8 量化、异步推理流水线等优化方向的可行性与预期收益，
为同类嵌入式机器人系统的性能优化与软件架构设计提供实验依据和工程参考。

\subsection{研究目标}
\begin{enumerate}[leftmargin=2em]
  \item 建立跨层次（AI框架层 / 运行时层 / OS内核层 / 硬件I/O层）性能分析模型；
  \item 端到端量化感知、推理、执行各阶段性能开销；
  \item 定位主要瓶颈，为系统优化与软件架构改进提供实验依据。
\end{enumerate}

\subsection{主要贡献}
\begin{itemize}[leftmargin=2em]
  \item 提出覆盖 AI 框架到内核的四层性能分析模型；
  \item 通过 ftrace/torch.profiler/perf 多工具联合剖析，量化各阶段时间开销；
  \item 揭示 strace 对系统调用测量带来的 9$\times$ 时间放大问题，验证 ftrace 的必要性；
  \item 基于量化结果，系统梳理优化路径及其成本收益。
\end{itemize}

\section{相关工作}

\subsection{机器人操作系统与通信框架}

\textbf{ROS2}：机器人操作系统（Robot Operating System 2）是 ROS 的下一代版本，
以 DDS（Data Distribution Service）作为底层通信中间件，支持节点间发布-订阅与服务调用。
ROS2 引入了节点生命周期管理、QoS 策略（可靠性/持久性/实时性配置）等企业级特性，
支持跨机器跨平台分布式部署，并可通过 \texttt{rclcpp} / \texttt{rclpy} 接入 C++ 或 Python 节点。
其通信延迟在局域网内通常为毫秒级，但节点调度仍依赖 Linux CFS 调度器，无硬实时保证。
ROS2 主要运行于 Ubuntu（官方支持 22.04/24.04），需配合 Linux 内核使用，不支持 Windows/macOS 原生部署。

\textbf{LCM}（Lightweight Communications and Marshalling）：由 MIT 开发的高性能消息传递库，
设计目标是为实时系统提供低延迟、零拷贝的消息序列化与传输能力。
LCM 使用 UDP 单播/广播而非 TCP，支持用户态消息队列（类型声明 + 字节流），
省去了 IDL 编译步骤，延迟可低至数微秒。
但 LCM 仅提供消息传递层，不含控制回路、参数服务器或 RViz 等生态工具，生态远不如 ROS2。

\subsection{开源机器人硬件平台}

\textbf{SO-101}：HuggingFace LeRobot 配套的六自由度机械臂，
结构件可 3D 打印，成本低廉，与 LeRobot 框架深度绑定，是本文实验平台的直接来源。

\textbf{OpenManipulator}：ROBOTIS 推出的开源机械臂，提供配套的 ROS2 集成包，
硬件图纸与代码全开源，适合教学和原型开发，支持 MoveIt! 运动规划。

\textbf{TurtleBot4}：开源移动机器人的事实标准，最新版本与 ROS2 生态深度集成，
配备 OAK-D 深度相机，适用于室内导航与移动操作任务。

\textbf{Berkeley Humanoid Lite}：UC Berkeley HybridRobotics 实验室开发的 3D 打印小型人形机器人，
为开源人形机器人平台提供了新思路。

\subsection{机器人策略模型}

机器人策略模型分为两大类：

\textbf{视觉-语言-动作模型（VLA）}：将视觉感知、语言理解和动作执行统一到端到端网络中，
通过大规模预训练实现开放场景泛化能力。
代表工作包括：
\begin{itemize}[leftmargin=2em]
  \item \textbf{RT-2}：Google DeepMind 提出，将视觉-语言模型（VLM）输出直接映射为机器人动作 token；
  \item \textbf{OpenVLA}：开源 VLA 模型，支持多任务泛化，参数量 7B，可在消费级 GPU 部署；
  \item $\boldsymbol{\pi_0}$：Physical Intelligence 提出，以流匹配（Flow Matching）为核心动作生成机制，
        在双臂协作任务上表现优异。
\end{itemize}

\textbf{模仿学习策略}：通过专家演示数据训练策略网络，不依赖大规模预训练。
代表工作包括：
\begin{itemize}[leftmargin=2em]
  \item \textbf{ACT}（Action Chunking with Transformers）：CVAE 编解码器 + Transformer 动作预测，
        支持动作分块（chunk\_size）和 time ensemble，适合桌面精细操作任务；
  \item \textbf{Diffusion Policy}：将动作生成建模为扩散过程，适合多模态动作分布。
\end{itemize}

\textbf{对比}：VLA 泛化能力强但计算量巨大（7B+ 参数），难以在嵌入式 ARM 平台实时运行；
ACT/Diffusion Policy 参数量较小（ResNet18 + Transformer，约 50M 参数），可在树莓派 5 上运行，
但推理延迟仍是主要瓶颈（2000--4000 ms/次）。

\subsection{嵌入式边缘 AI 推理}

在资源受限的嵌入式平台上部署深度学习模型，需要针对硬件特性进行优化：

\textbf{TFLite}：Google 推出的移动端推理框架，支持 FP16/INT8 量化、
ARM NEON SIMD 加速，广泛用于边缘设备推理。

\textbf{ONNX Runtime}：微软推出的跨平台推理引擎，支持多种执行提供者（CPU/ CUDA / TensorRT），
在 ARM 上通过 OpenMP 和 NEON 加速，延迟优于 TFLite 约 1.5--2×。

\textbf{PyTorch Mobile}：PyTorch 的移动端部署方案，
支持 TorchScript 导出和 CPU 优化，但相比 ONNX Runtime 在 ARM 上的优化程度较低。

\textbf{树莓派 5（Cortex-A76）计算能力}：4 核 4 GB LPDDR4X，
双发射 SIMD 单元支持 NEON，矩阵乘法（GEMM）理论峰值约 50 GFLOPS（FP32），
实际推理 ACT 模型约 30--50 GFLOPS 利用率，实测推理耗时 2000--4000 ms/次。
作为对比，RT-2（7B 参数）在树莓派 5 上几乎无法运行，VLA 模型需在 GPU 平台上部署。

\subsection{系统级 Profiling 技术}

机器人系统的性能分析需要跨层次工具链：

\begin{itemize}[leftmargin=2em]
  \item \textbf{strace}：通过 \texttt{ptrace} 拦截系统调用，开销极大（9× 时间放大），仅用于定性探索；
  \item \textbf{ftrace}：内核态 tracepoint，开销极低（纳秒级），适合生产环境测量；
  \item \textbf{perf}：硬件性能计数器（PMC），可测量 IPC、Cache-miss 等微架构指标；
  \item \textbf{torch.profiler}：PyTorch 算子级 profiling，分解前向/反向传播各算子耗时。
\end{itemize}

已有机器人系统性能分析工作多聚焦于单一层次（模型压缩、内核调度或通信延迟），
缺乏从 AI 框架层到内核层的跨层次联合剖析。

\subsection{工具链选型}

\textbf{strace vs ftrace 机制与开销对比}：

两者作用层次不同，开销来源也不同：

\begin{itemize}[leftmargin=2em]
  \item \textbf{strace}：通过 \texttt{ptrace} 系统调用在用户态拦截目标进程的每一次系统调用入口与出口。
        每次拦截都需要一次进程上下文切换（目标进程 $\to$ strace 进程），
        导致系统时间从 \textbf{2.20 s 放大至 20.12 s（约 9×）}。
        其开销与系统调用频率正相关，频繁调用时干扰极大，不可用于生产级测量。

  \item \textbf{ftrace}：通过 Linux 内核 tracepoint / \texttt{trace\_printk()} 在内核态直接写入 ring buffer，
        \textbf{插桩本身开销极低}（纳秒级，无进程切换）。
        但将 ring buffer 内容导出至用户态文件（\texttt{cat /sys/kernel/debug/tracing/trace > log}）
        是一次批量文件 I/O，当 event 密集时，\textbf{导出过程本身会消耗额外 CPU 和内存带宽}，
        且导出期间 ring buffer 可能覆盖旧数据（丢失事件）。

  \item \textbf{本文策略}：以 ftrace 插桩数据为主要内核态度量依据；
        strace 仅用于初期定性探索系统调用分布，不计入最终时序数据。
\end{itemize}

\section{异步化可行性调研}

\subsection{摄像头异步读}

LeRobot 摄像头采集通过 OpenCV 后台线程实现异步：主线程调用
\texttt{retrieveFrame()} 时，后台线程已在 \texttt{pselect6()} 等待下一帧。
\begin{itemize}[leftmargin=2em]
  \item \textbf{当前状态}：已是默认实现，无需改造；
  \item \textbf{可优化点}：MJPEG 解码（10--30 ms/帧）为 CPU 密集瓶颈，
        切换为 YUYV 格式可直接消除解码开销；
  \item \textbf{结论}：摄像头异步读已实现，当前主要矛盾在解码格式而非异步化。
\end{itemize}

\subsection{推理异步}

LeRobot 支持双进程推理架构（主进程 + 推理进程，通过 gRPC 通信）：
\begin{itemize}[leftmargin=2em]
  \item \textbf{当前状态}：已实现但未启用（当前配置下 stall 不可避免）；
  \item \textbf{不 stall 条件推导}：
        \[
          N \geq 2 \times T_{\text{infer}} \times F
        \]
        代入参数（$T_{\text{infer}} = 2\,\text{s}$，$F = 30\,\text{fps}$，$N = 100$）：
        \[
          \text{需要}\; N \geq 120, \quad \text{当前}\; N = 100 < 120 \quad \Rightarrow \text{stall 不可避免}
        \]
  \item \textbf{结论}：配合 INT8 量化（预期 $T_{\text{infer}}$ 降至 0.5--1 s）后，
        可将不 stall 所需 $N$ 降至 30--60，当前 $N=100$ 可满足条件，推理异步方可启用。
\end{itemize}
\src{async\_inference\_推理执行流水线分析.md}

\subsection{舵机异步读}

舵机位置读取通过 \texttt{sync\_read} 实现（半双工写请求 + 读回包），异步化改造存在以下难点：
\begin{itemize}[leftmargin=2em]
  \item \textbf{总线半双工约束}：读写操作必须串行，异步并发写入会导致总线冲突；
  \item \textbf{安全限幅链路}：\texttt{send\_action()} 需先读取当前位置（read-modify-write），
        若读操作异步化，则无法保证限幅判断的时效性；
  \item \textbf{结论}：在当前半双工总线协议下，舵机异步读难以实现，不建议作为优化方向。
\end{itemize}
\src{Lerobot内舵机读写完整链路.md，motors\_bus异步读写设计分析.md}

\subsection{舵机异步写}

类似异步读，舵机指令下发（\texttt{sync\_write}）的异步化存在根本性约束：
\begin{itemize}[leftmargin=2em]
  \item \textbf{总线半双工冲突}：异步并发写会导致总线冲突，指令帧损坏；
  \item \textbf{安全限幅链路断裂}：\texttt{send\_action()} 中的 read-modify-write 操作依赖同步读返回的实时位置，
        若写操作先于读完成，限幅判断将基于过期数据，可能导致机械臂碰撞；
  \item \textbf{相关工作}：调研现有机器人控制系统异步执行架构~\cite{async_robot}，
        均在支持全双工或具备独立控制通道的硬件平台上实现，不适用于半双工 TTY 总线；
  \item \textbf{结论}：舵机异步写在当前硬件平台上不可行，亦非必要（写操作本身仅 1--3 ms）。
\end{itemize}

\section{总结与展望}

\subsection{主要工作总结}
\begin{enumerate}[leftmargin=2em]
  \item 建立了覆盖 AI 框架到内核的四层性能分析模型；
  \item 通过 Python 日志 + ftrace + torch.profiler 多工具联合，端到端量化了各阶段开销；
  \item 揭示 ACT 推理（2000--4000 ms）为主瓶颈，系统调用 overhead 可忽略；
  \item 推导了异步推理流水线不 stall 的数学条件，指出当前参数配置的局限性；
  \item 系统梳理了各优化路径的可行性与成本收益；
  \item 对感知、推理、执行三阶段进行了从 Python 到 kernel 的全链路 CallStack 源码分析；
  \item 调研了各阶段异步化可行性，论证了舵机异步读/写的不可行性及其原因。
\end{enumerate}

\subsection{研究局限性}
\begin{itemize}[leftmargin=2em]
  \item 仅在 SO101 单臂场景下测试，未覆盖双臂或移动机器人；
  \item 未实施量化等优化方案并验证，结论停留在分析层面；
  \item 无 GPU 对比基线，难以量化 CPU 推理与加速器的差距。
\end{itemize}

\subsection{未来工作}
\begin{itemize}[leftmargin=2em]
  \item INT8 量化 + ONNX Runtime 实验，验证推理加速收益；
  \item 扩展至多任务场景（更多抓取类别）与多机器人平台；
  \item 探索面向机器人负载特征的实时操作系统设计。
\end{itemize}
