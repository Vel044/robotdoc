\begin{center}
  {\zihao{-2}\heiti\textbf{LeRobot 平台感知—推理—执行流程的\\[4pt]系统级性能剖析研究}}\\[1.2em]
  {\zihao{3}\heiti\textbf{摘\quad 要}}
\end{center}

\vspace{1em}
\zihao{-4}\songti
\mytodo{正式摘要待实验完成后撰写，约300-500字。}

本文以树莓派5（ARM Cortex-A76，无GPU）为目标平台，对 LeRobot 框架下 ACT 机器人学习策略的感知—推理—执行三阶段时序特征进行系统级剖析。
通过在 Python 应用层、PyTorch 运行时层、Linux 内核层及硬件I/O层的多工具联合测量（Python logging / torch.profiler / ftrace / strace），
端到端量化了各阶段的时间开销，定位了主要性能瓶颈，
并从算法、运行时、系统架构三个维度梳理了可行的优化路径。

实验结果表明，ACT 单次推理耗时 2000--4000 ms，占总循环时间的 99\% 以上，是系统主瓶颈；
观测和执行阶段开销极低（各约 5 ms），系统调用本身的开销可忽略不计，
排除了 I/O 路径为主要瓶颈的假设。
在 chunk\_size=100 配置下，系统实际帧率约 18--19 fps，距 30 fps 目标仍有明显差距，
瓶颈根源在于 ARM CPU 矩阵乘法（GEMM）计算能力不足，而非感知复杂度或任务类型。

    System-Level Performance Profiling of\\[4pt]
    Perception-Inference-Execution Pipeline\\[4pt]
    in the LeRobot Framework}\\[1.2em]
  {\zihao{3}\rmfamily\bfseries Abstract}
\end{center}

\vspace{1em}
\zihao{-4}\rmfamily
\mytodo{English abstract to be finalized after experiments complete, approximately 300 words.}

This paper presents a system-level performance analysis of the ACT (Action Chunking with Transformers) policy
running on a Raspberry Pi 5 (ARM Cortex-A76, no GPU) within the LeRobot framework.
By combining multi-layer profiling tools across the Python application layer, PyTorch runtime layer,
Linux kernel layer, and hardware I/O layer (Python logging, torch.profiler, ftrace, strace),
we quantify the end-to-end time distribution across perception, inference, and execution stages,
identify the primary bottleneck, and systematically evaluate feasible optimization paths.

Results show that a single ACT inference takes 2000--4000\,ms, accounting for over 99\% of the total
loop time, making it the dominant bottleneck.
Perception and execution stages contribute only $\approx$5\,ms each;
system call overhead is negligible, ruling out I/O as a bottleneck.
At chunk\_size=100, the system achieves $\approx$18--19\,fps against a 30\,fps target.
The root cause is insufficient ARM CPU matrix-multiplication (GEMM) throughput,
independent of task visual complexity or object count.

\vspace{1.5em}

LeRobot; ACT; performance profiling; ARM inference; chunk\_size; perception-inference-execution}

\section{绪论}

\subsection{研究背景与意义}
\begin{itemize}[leftmargin=2em]
  \item 协作机器人与 VLA 模型快速发展，软件栈日趋复杂；
  \item 主流机器人系统基于通用宏内核构建，与"感知—推理—执行"紧耦合运行特征存在明显差异；
  \item 引入 ACT 等 AI 推理框架后，系统性能瓶颈与资源消耗机理缺乏系统性分析。
\end{itemize}

\begin{enumerate}[leftmargin=2em]
  \item 建立跨层次（AI框架层 / 运行时层 / OS内核层 / 硬件I/O层）性能分析模型；
  \item 端到端量化感知、推理、执行各阶段性能开销；

\begin{itemize}[leftmargin=2em]
  \item 提出覆盖 AI 框架到内核的四层性能分析模型；
  \item 通过 ftrace/torch.profiler/perf 多工具联合剖析，量化各阶段时间开销；

\section{系统级性能剖析结果}

\subsection{感知阶段}

\subsubsection{摄像头路径}

调用链：\texttt{cv2.VideoCapture.read()} $\to$ \texttt{grabFrame}（VIDIOC\_DQBUF）
$\to$ \texttt{retrieveFrame}（libjpeg 解码）。

\begin{itemize}[leftmargin=2em]
  \item pselect6 行为：摄像头后台线程阻塞 \textbf{23--32 ms}，符合 30 fps 帧间隔物理极限；
  \item MJPEG 解码耗时：\textbf{10--30 ms/帧}（libjpeg CPU 密集，是摄像头路径主要开销）；
  \item 整体摄像头贡献观测耗时约 \textbf{1.9 ms}（主线程视角，后台异步）。
\end{itemize}
\src{pselect6最终结果.md，OpenCV视频流数据流分析.md}

\subsubsection{舵机路径}

调用链：\texttt{motors\_bus.sync\_read()} $\to$ \texttt{GroupSyncRead}
$\to$ \texttt{PortHandler} $\to$ \texttt{/dev/ttyACM0}。

\begin{itemize}[leftmargin=2em]
  \item Sync Read 包格式：\texttt{[0xFF 0xFF 0xFE LEN 0x82 ADDR LEN ID$_1$..ID$_6$ CHK]}；
  \item Present\_Position 寄存器地址 56，sign-magnitude 编码（bit15 为方向位）；
  \item 耗时：\textbf{1.2 ms/帧}，开销可忽略。
\end{itemize}
\src{observation读取ACM0数据完整流程.md，Lerobot内舵机读写完整链路.md}

  \item futex 调用：\textbf{64,569 次 / 40 s episode}，总等待时间 3,252 s（27线程并行累计）；
  \item op=0（FUTEX\_WAIT）占多数，op=9（FUTEX\_WAIT\_BITSET）用于 ATen 线程池；
  \item ARM Cortex-A76 使用 OpenMP 并行，NEON SIMD 加速 GEMM，但计算量仍是瓶颈。
\end{itemize}
\src{futex最终结果.md，ACT推理.md}

\subsection{执行阶段}

\subsubsection{sync\_write 路径}
\begin{itemize}[leftmargin=2em]
  \item 调用链：\texttt{motors\_bus.sync\_write()} $\to$ \texttt{GroupSyncWrite.txPacket()}
        $\to$ \texttt{write()} $\to$ TTY $\to$ USB bulk transfer；
  \item 无回包，半双工 TTY 总线，耗时 \textbf{1--3 ms}；
  \item 异步写不可行：总线半双工冲突 + 安全限幅 read-modify-write 链路断裂。
\end{itemize}
\src{USB串口写路径.md，motors\_bus异步读写设计分析.md}

\subsubsection{异步推理流水线分析}

不 stall 条件推导：
\[
  N \geq 2 \times T_{\text{infer}} \times F
\]

代入当前参数（$T_{\text{infer}} = 2\,\text{s}$，$F = 30\,\text{fps}$，$N = 100$）：
\[
  \text{需要}\; N \geq 120, \quad \text{当前}\; N = 100 < 120 \quad \Rightarrow \text{stall 不可避免}
\]
\begin{itemize}[leftmargin=2em]
  \item stall 周期：20 帧卡顿 / 70 帧正常，循环出现；
  \item time ensemble 失效：chunk B 到达时前 50 帧已过期被丢弃；
  \item 增大 chunk\_size 会导致动作质量下降，非可行方案。
\end{itemize}

\section{性能瓶颈分析与优化建议}

\subsection{瓶颈归因}
\begin{itemize}[leftmargin=2em]
  \item \textbf{主瓶颈}：ACT 推理（ARM CPU 矩阵乘法），2000--4000 ms/次；
  \item \textbf{次瓶颈}：MJPEG 解码（10--30 ms/帧，可替换为 YUYV 消除）；
  \item \textbf{非瓶颈确认}：系统调用本身（pselect6 421 ns，futex $<$1 $\mu$s，均远低于推理耗时）。
\end{itemize}

\subsection{优化方向分析}

\begin{center}
\begin{tabular}{p{3.5cm}p{2.8cm}cc}
  \toprule
  优化方向 & 预期收益 & 可行性 & 复杂度 \\
  \midrule
  INT8/FP16 量化          & 推理降 2--4$\times$ & 高 & 中 \\
  ONNX Runtime 后端       & 推理降 1.5--2$\times$ & 高 & 低 \\
  异步推理（gRPC 双进程）  & 消除 stall（需配合量化） & 中 & 高 \\
  MJPEG $\to$ YUYV        & 解码降至 0 & 高 & 低 \\
  异步舵机写              & \textbf{不可行} & \textcolor{red}{×} & — \\
  \bottomrule
\end{tabular}
\end{center}

\subsection{软件架构改进方向}
\begin{itemize}[leftmargin=2em]
  \item 推理模块解耦（独立进程 + gRPC），避免 GIL 与主循环相互干扰；
  \item 摄像头格式统一为 YUYV，消除 libjpeg CPU 解码路径；
  \item 面向机器人负载的内核优化方向：实时调度器（SCHED\_FIFO）、USB 驱动优先级提升。
\end{itemize}

\section{总结与展望}

\subsection{主要工作总结}
\begin{enumerate}[leftmargin=2em]
  \item 建立了覆盖 AI 框架到内核的四层性能分析模型；
  \item 通过 Python 日志 + ftrace + torch.profiler 多工具联合，端到端量化了各阶段开销；
  \item 揭示 ACT 推理（2000--4000 ms）为主瓶颈，系统调用 overhead 可忽略；
  \item 推导了异步推理流水线不 stall 的数学条件，指出当前参数配置的局限性；
  \item 系统梳理了各优化路径的可行性与成本收益。
\end{enumerate}

\subsection{研究局限性}
\begin{itemize}[leftmargin=2em]
  \item 仅在 SO101 单臂场景下测试，未覆盖双臂或移动机器人；
  \item 未实施量化等优化方案并验证，结论停留在分析层面；
  \item 无 GPU 对比基线，难以量化 CPU 推理与加速器的差距。
\end{itemize}

\subsection{未来工作}
\begin{itemize}[leftmargin=2em]
  \item INT8 量化 + ONNX Runtime 实验，验证推理加速收益；
  \item 扩展至多任务场景（更多抓取类别）与多机器人平台；
  \item 探索面向机器人负载特征的实时操作系统设计。
\end{itemize}
